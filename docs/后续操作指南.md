# domain-test 后续操作指南

本文从「日常开发」到「发版、分发、运维」列出建议步骤，可与根目录 [README.md](../README.md) 对照使用。

---

## 一、日常开发（自己或团队改代码）

1. **进入项目并激活环境**（venv 或 conda 任选其一，与 README 一致）。
2. **可编辑安装**（改代码立即生效）：  
   `pip install -e ".[dev]"`  
   若已装过，仅需 `git pull` 后一般无需重装，除非 `pyproject.toml` 依赖有变。
3. **准备配置**：程序有**内置默认**；任选 **`.env` / `export ROUTER_*`**，或 **`--config ./config.yaml`**（仓库根 `config.yaml` 为可选模板）。勿提交含真实密码的 `.env`。
4. **准备域名列表**：如 **`domains.txt`**。
5. **在任意目录执行**（`-d` 路径相对 cwd）：  
   ```bash
   domain-test -d domains.txt --no-headless
   ```  
   不写 **`-o`** 时，报告在**当前目录**下的 **`run_<时间戳>/`**。
6. **看结果**：该目录内 **xlsx**、**png**；根据结果改环境变量、`--config` 文件或业务逻辑。

---

## 二、版本号与变更记录（发版前必做）

1. 打开 **[pyproject.toml](../pyproject.toml)**，将 **`[project] version`** 改为新版本（如 `0.1.0`）。  
2. （建议）同步修改 **[domain_test/__init__.py](../domain_test/__init__.py)** 中的 **`__version__`**，与 `pyproject.toml` 一致。  
3. （可选）在 README 或单独 CHANGELOG 中写一句本版变更说明。  
4. **提交 Git**：  
   ```bash
   git add pyproject.toml domain_test/__init__.py README.md docs/
   git commit -m "chore: bump version to 0.1.0"
   git push origin main
   ```

## 三、发布到 PyPI（官方索引）

### 3.1 一次性准备

1. 在 [pypi.org](https://pypi.org) 注册账号。  
2. 创建 [API Token](https://pypi.org/manage/account/token/)（权限覆盖本项目或整个账号，按 PyPI 说明选择）。  
3. 确认 **`pyproject.toml` 里的 `name`**（当前为 **`domain-test`**）在 PyPI **未被占用**；若被占用需改名并全局替换文档中的安装命令。

### 3.2 本地构建并手动上传（不依赖 GitHub）

```bash
cd /path/to/domain-test
pip install -e ".[dev]"
rm -rf dist build *.egg-info
python -m build
twine check dist/*
export TWINE_USERNAME=__token__
export TWINE_PASSWORD=pypi-你的token
twine upload dist/*
```

上传成功后，他人可执行：`pip install "domain-test==0.1.0"`（版本号与实际上传一致）。

### 3.3 用 GitHub Actions 自动上传（已配置工作流）

1. 在 GitHub：**Settings → Secrets and variables → Actions** → 新建 **`PYPI_API_TOKEN`**，值为上述 token。  
2. 本地 **`version`** 与 **`__version__`** 已改好并提交、推送。  
3. 打标签并推送（示例）：  
   ```bash
   git tag v0.1.0
   git push origin v0.1.0
   ```  
   或到 **Actions → Publish to PyPI → Run workflow** 手动触发。  
4. 在 Actions 页面查看是否绿色通过；失败时看日志（常见：token 无效、版本号已存在、twine check 失败）。

## 四、发布到私有源（Nexus / DevPI / 自建 PyPI）

1. 同样先 **`python -m build`** 得到 **`dist/*`**。  
2. 按你们制品库文档执行 **`twine upload`**，通常带 **`--repository-url`**、**`-u` / `-p`**。  
3. 使用方配置 **`pip install --index-url https://... simple/`** 或 **`pip.conf` 的 extra-index-url**（README 中有示例命令）。

---

## 五、给同事 / 客户用（不发 PyPI 时）

1. **Git**：对方 `git clone` → 建环境 → `pip install -r requirements.txt` → **`pip install -e .`**。  
2. 把 **`.env.example`**、**`domains.example.txt`**、可选 **`config.yaml` 模板** 的使用说明发给对方；敏感项只用环境变量或对方本地 `.env`。  
3. 说明：**wheel 内含程序 `.py` 源码**（可阅读实现）；**不含**对方本机的 `config.yaml`；对方用 **环境变量** 或 **`--config`** 提供路由器等配置。

## 六、仓库与文件卫生

| 建议 | 说明 |
|------|------|
| **不要提交** | `.env`、含密码的 `config.yaml` 变体、业务产生的 **`run_*`**、大 xlsx |
| **可删除** | `build/`、`dist/`、`*.egg-info/`（会再生）、本地 `run_*` 目录 |
| **应提交** | 源码、`pyproject.toml`、`README.md`、`LICENSE`、`docs/`、`.github/workflows/` |

## 七、常见问题排查顺序

1. **配置校验失败**：设置 **`ROUTER_HOST` 等** 或 **`--config`** 指向有效 YAML；确认 **`pyproject.toml` 里 `[tool.setuptools.package-data]`** 已包含 **`domain_test/builtin_config.yaml`**（否则从 wheel 安装会读不到内置默认）。  
2. **`域名列表文件不存在`**：`-d` 路径相对 **cwd**，先 `pwd` / `ls` 确认。  
3. **SSH / NAT 失败**：检查 `ROUTER_*`、防火墙、`router.ssh_encoding`。  
4. **Chrome 无法启动**：安装 Chrome、设置 `CHROME_PATH` 或 `browser.chrome_executable`。  
5. **PyPI 上传失败**：版本是否已存在、token 是否正确、是否先 **`twine check`**。

## 八、可选的下一步（非必须）

- 加 **`CHANGELOG.md`**、**语义化版本**规范。  
- 加 **单元测试**（`pytest`）与 CI 里 **`pytest` + `build`**。  
- **退出码**：部分失败用非 0 退出，便于 CI。  
- **JSON 报告**：除 Excel 外再写一份机器可读结果。
